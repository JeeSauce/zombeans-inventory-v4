## Phase 3 additions

| ID    | Date       | Decision                                                                                                                                                                                                                                                                                                                                 | Rationale                                                                                                                                                                                                                                                                                     |
| ----- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A-021 | 2026-07-11 | Weighted-average cost (`inventory_items.weighted_avg_cost`) is a **global, per-item** value (not per-branch). `post_purchase_receipt()` recomputes it by reading `inventory_balances.qty_on_hand`, which is branch-scoped; this is self-consistent for the single-branch MVP (Phase 2 seeds only one branch that holds raw ingredients). | Spec calls for a single weighted-average per item. A true multi-branch weighted-average needs either a global on-hand rollup or a per-branch cost column — deferred until a second stock-holding branch exists; revisit before then.                                                          |
| A-022 | 2026-07-11 | `post_supplier_return()` removes quantity at the **lot's recorded cost** (FEFO-selected or explicitly targeted lot) and does **not** recompute `weighted_avg_cost` — a return of stock already blended into the average doesn't retroactively unblend it.                                                                                | Matches standard weighted-average inventory accounting: returns are valued at the cost they were carried at, not re-derived; keeps the average stable and auditable.                                                                                                                          |
| A-023 | 2026-07-11 | `post_purchase_receipt()` posts stock-in at the **expected PO unit cost** captured on the purchase-order line at approval time. If a supplier's invoice later differs from the PO cost, reconciling that difference is a Super-Admin follow-up (a manual correcting entry), not automated this phase.                                    | No invoice-capture/matching workflow exists yet; posting at the agreed PO cost keeps the ledger consistent with what was actually approved, and avoids inventing an invoice-diff UI this phase.                                                                                               |
| A-024 | 2026-07-11 | Added `supplier_price.write` to the permission catalog, granted to `super_admin` only (not `branch_manager`, despite branch managers holding `purchase.create`).                                                                                                                                                                         | Supplier pricing is SENSITIVE data (rule 4); only the role that can already see every cost should be able to edit the price history that feeds auto-filled PO line costs.                                                                                                                     |
| A-025 | 2026-07-11 | The posting RPCs `post_purchase_receipt()` / `post_supplier_return()` are `SECURITY DEFINER` but do **not** call an in-function `has_permission()` check; enforcement is via RLS on the receipt/return row (the caller must already be able to see the row it's posting) plus `requirePermission()` in the calling server action.        | Deliberate deferral — the server action is the only caller today and already gates on `purchase.receive`/`supplier.write`; an in-function check is straightforward to add but was scoped out to keep Phase 3 to its planned surface. Revisit if a second caller (e.g. a future API) is added. |
